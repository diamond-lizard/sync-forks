* Overview

~sync-forks~ reads a JSON file containing GitHub fork metadata, identifies
forks that are behind their upstream repositories, and brings them up to date
using the GitHub merge-upstream API.

Forks are classified into three categories:

- *Behind only* (syncable): behind upstream with no local commits ahead.
  These are synced automatically.
- *Ahead and behind* (skipped): have local commits that diverge from
  upstream.  These are reported but not synced to avoid overwriting local
  work.
- *Not behind* (ignored): already up to date or ahead only.

* Requirements

- Python >= 3.12
- [[https://github.com/astral-sh/uv][uv]] (for dependency management)
- [[https://www.passwordstore.org/][pass]] (password manager, for GitHub token retrieval)
- A GitHub fine-grained personal access token (FGPAT) stored in ~pass~ at
  ~github.com/fgpat/repos-rw~ with the following repository permissions:
  - *Contents*: Read and write
  - *Metadata*: Read (automatically selected)
  - *Workflows*: Read and write (required when upstream repos contain
    ~.github/workflows/~ files)

* Installation

#+begin_src shell
uv pip install -e .
#+end_src

* Usage

#+begin_src
sync-forks [OPTIONS] FILENAME
#+end_src

~FILENAME~ is a JSON file containing fork metadata, or ~-~ to read from
stdin.

** Options

| Option      | Description                                      |
|-------------+--------------------------------------------------|
| ~--dry-run~ | Show what would be done without making any changes. No API calls are made and no token is retrieved. |
| ~--quiet~   | Suppress informational output.  Errors and abort messages are still printed. |
| ~--help~    | Show usage information and exit.                 |

** Examples

Dry-run to preview what would be synced:

#+begin_src shell
bin/sync-forks --dry-run forks.json
#+end_src

Sync all behind-only forks:

#+begin_src shell
bin/sync-forks forks.json
#+end_src

Read from stdin:

#+begin_src shell
cat forks.json | bin/sync-forks -
#+end_src

Quiet mode (errors only):

#+begin_src shell
bin/sync-forks --quiet forks.json
#+end_src

* Input Format

The input JSON must be an object with a ~"results"~ array.  Each entry with
~"status": "fork"~ must have ~repo_url~, ~ahead_by~, and ~behind_by~ fields:

#+begin_src json
{
  "results": [
    {
      "status": "fork",
      "repo_url": "https://github.com/owner/repo",
      "ahead_by": 0,
      "behind_by": 5
    }
  ]
}
#+end_src

Entries with a status other than ~"fork"~ are ignored.  Fork entries missing
required fields are reported on stderr and skipped.

* Exit Codes

| Code | Meaning                                                  |
|------+----------------------------------------------------------|
|    0 | Success, nothing to do, or partial success               |
|    1 | Application error (token failure, rate limit exhaustion, host error threshold exceeded, keyboard interrupt) |
|    2 | Usage error (missing arguments, invalid options)         |

* Error Handling

- *Rate limiting*: Detected via ~x-ratelimit-remaining~ header or
  ~"secondary rate limit"~ in the response body.  Retries up to 10 times
  with exponential backoff.  Aborts the run if retries are exhausted.
- *5xx server errors*: Retried once after a 5-second delay.
- *Per-host error threshold*: After 5 non-rate-limit errors from a single
  host, the run is aborted.  HTTP 409 Conflict does not count toward this
  threshold.
- *Keyboard interrupt*: Prints a summary of progress so far, prints
  "Interrupted.", and exits with code 1.  No traceback is shown.

* Troubleshooting

** HTTP 401 on all repositories

All repos fail immediately with ~HTTP 401~ and the run aborts after 5
errors.  This means the GitHub token is invalid.

- The token stored in ~pass~ at ~github.com/fgpat/repos-rw~ may have
  expired.  Fine-grained PATs have expiration dates.
- The token may have been revoked or rotated without updating ~pass~.
- Verify the token is still valid:

#+begin_src shell
pass show github.com/fgpat/repos-rw | head -1 | gh auth status --with-token
#+end_src

If expired, mint a new token and update it in ~pass~.

** HTTP 422 on some repositories (merge-upstream failed)

Some repos sync successfully while others fail with ~HTTP 422~.  The error
message from GitHub will appear after the status code, for example:

: o/r: HTTP 422: refusing to allow a Personal Access Token to create or
: update workflow `.github/workflows/ci.yml` without `workflow` scope

Common causes:

- *Missing Workflows permission*: The upstream commits contain changes to
  ~.github/workflows/~ files, but the token lacks the *Workflows: Read and
  write* permission.  Add this permission to the token.
- *Missing repository access*: The token is configured with "Selected
  repositories" and some repos are not in the selection.  Add the missing
  repos to the token's repository access list.

** HTTP 409 Conflict

The fork has diverged from upstream (commits both ahead and behind) in a
way that cannot be fast-forwarded.  This typically means someone pushed
commits directly to the fork's default branch.  These errors do not count
toward the per-host error threshold, so they will not cause an abort.

Resolve by manually merging or rebasing the fork.

** Rate limit exhaustion (aborted after retries)

The run is aborted with a message about rate limit retries being exhausted.
This happens when too many API calls are made in a short period.

- Wait for the rate limit window to reset (usually 1 hour for
  authenticated requests).
- If this happens frequently, consider splitting the input file into
  smaller batches.
